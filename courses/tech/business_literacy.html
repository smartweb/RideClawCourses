01为什么技术人员要懂商业？

很多技术同学认为："我只管写代码，商业是老板的事。"但在出行行业，不懂商业的技术决策，往往会导致华丽的废品。

避免自嗨式开发

你做的功能用户真的付费吗？99%的「酷炫功能」对商业结果没有影响，反而增加维护成本。

理解技术优先级

为什么先接A供应商而不是B？为什么容忍5秒查询延迟？商业上的ROI决定技术投入的排序。

与业务团队同频

当产品经理说「GMV」「转化率」「库存满足率」时，你能听懂背后的技术含义。

发现创新机会

商业痛点往往藏在技术细节里。真正好的创新，是技术与商业的交汇点。

"伟大的工程师不是只写优雅代码的人，而是能用技术解决真实商业问题的人。"

03机票业务：产品、分销与收益管理

机票可能是出行行业里最复杂的商品：同一架飞机上的座位，价格可能相差10倍。为什么？

机票的特殊性

  • 库存极易过期（起飞后座位价值归零）
  • 价格动态变化（每秒都可能调整）
  • 产品高度标准化（座位+时刻+服务）
  • 退改签规则复杂（不同舱位不同政策）
  • 分销依赖GDS/NDC协议

技术同学的关注点

  • 库存查询性能（毫秒级响应）
  • 缓存一致性（价格不能过期）
  • 订单状态机（PNR/出票/改签/退票）
  • 多供应商聚合（GDS+NDC+直连）
  • 辅营产品打包（行李+选座+餐食）

收益管理（Revenue Management）

航空公司通过收益管理系统决定每个舱位卖多少钱。核心算法是：在起飞前把座位卖给「愿意付最高价的旅客」。

# 简化版收益管理逻辑
expected_revenue = price * probability_of_sale
# 目标: 在起飞前最大化所有座位的 expected_revenue 之和
# 涉及: 需求预测、竞品价格、历史数据、节假日、事件

舱位（Class）

同一物理座位，因退改签规则、里程累积、行李额度不同而形成不同舱位代码（Y/B/M/H/K...）。

运价（Fare Basis）

由字母和数字组成的代码，定义了一个票价产品的全部规则。技术人员解析它来判断退改签政策。

PNR

旅客订座记录，6位字母数字组合。一次行程的所有乘客、航班、座位、特殊服务都在里面。

04酒店业务：库存、价格与渠道管理

酒店比机票简单一些，但也有自己的复杂性：每个房间的每晚价格独立，渠道众多，库存管理必须精细到「房态」。

房态（Availability）

某天某房型是否可售。必须实时同步到所有渠道，避免超售（Overbooking）或空置。

BAR价（Best Available Rate）

酒店官网最优价，通常是渠道价格的锚点。OTA价格不能低于BAR，否则违反Rate Parity。

渠道管理（Channel Management）

酒店同时向官网、携程、Booking.com、Expedia、TMC等渠道放库存。需要Channel Manager统一调控。

包房/切房

平台提前以固定价买断某段时间的库存，承担卖不完的风险，换取更高毛利。

05打车/地面交通：网络效应与调度系统

打车业务和前两者最大的不同：它不是卖库存，而是实时匹配供给（司机）和需求（乘客）。

核心商业特征

  • 双边网络效应：司机越多→等车越短→乘客越多→司机收入越高
  • 实时匹配：每笔订单都在几秒内完成供需匹配
  • 动态定价：高峰/雨天/偏远地区自动加价
  • 服务不可存储：司机空驶时间就是浪费
  • 地域性强：每个城市都是独立战场

技术同学的关注点

  • 供需预测算法（哪里会缺车？）
  • 派单策略（最近？最快？最顺路？）
  • 动态定价模型（如何平衡供需？）
  • 司机调度与热力图
  • ETA估算（预计到达时间）

关键公式：供需平衡

# 动态定价的核心逻辑
surgeMultiplier = f(supply_demand_ratio, wait_time, cancellation_rate)
# 当 surgeMultiplier > 1.0 时，用户看到加价
# 加价目的: 1)抑制需求 2)激励更多司机上线

对龙虾出行的启示：如果我们做接送机/包车，核心不是拥有车辆，而是建立可靠的调度系统和供应商网络。

06行业术语词典（技术视角）

以下是你在日常工作中最可能遇到的术语，以及它们背后的技术含义。

GDS

全球分销系统，航空业的「中央数据库」。

技术含义：Amadeus/Sabre/Travelport的API，通常使用EDIFACT老协议，查询费用按look-to-book计费。

NDC

新分销能力，航空零售化的技术标准。

技术含义：基于XML/JSON的REST API，支持更丰富产品展示、动态定价和辅营销售。

PMS / CRS

酒店物业管理系统 / 中央预订系统。

技术含义：PMS管单店运营，CRS管集团库存。我们通过API与它们同步房态和价格。

API聚合商

把多个供应商API封装成统一接口的公司。

技术含义：如Kyte、Duffel、Travelfusion，让我们用一套接口接入全球多家航司/酒店。

BSP

开账与结算计划，机票行业的资金清算系统。

技术含义：代理人出票后，通过BSP统一结算给航司。中国由CATA管理。

Look-to-Book

查询与预订比例。

技术含义：GDS/供应商按查询次数收费，缓存策略直接影响成本。健康值通常 < 500:1。

Commission / Rebate

佣金 / 返点，平台的收入来源。

技术含义：需要在订单系统中记录协议佣金率，并在结算时准确计算。

Voucher / Coupon

代金券 / 优惠券。

技术含义：涉及复杂的叠加规则、有效期、适用范围、风控防薅等技术逻辑。

07技术如何驱动商业价值

在出行行业，技术不是成本中心，而是利润杠杆。下面用具体例子说明技术与商业结果的关系。

搜索推荐算法

推荐准确度提升10% → 转化率提升2% → 年收入增加数百万。
对应技术：排序模型、用户画像、A/B测试

缓存与查询优化

Look-to-Book从800:1降到300:1 → GDS查询成本下降60%。
对应技术：Redis缓存、预计算、智能刷新策略

订单履约自动化

出票成功率从92%提升到99% → 客服工单下降50%。
对应技术：状态机、重试机制、异常告警、对账系统

动态打包引擎

机票+酒店打包销售 → 客单价提升33%，毛利更高。
对应技术：规则引擎、价格组合优化、实时库存校验

反欺诈风控

信用卡拒付率从3%降到0.5% → 年避免损失数百万。
对应技术：风控模型、设备指纹、行为分析、黑名单

客服自动化

AI客服处理80%标准问题 → 人力成本下降35%。
对应技术：NLP、RAG知识库、工单自动分类

"技术团队的KPI不应该只是系统可用性99.9%，而是「我们优化的系统，让公司多赚了多少钱 / 省了多少钱」。"

这些指标会出现在业务团队的周会上。理解它们的定义和计算方式，能帮助你做出更好的技术决策。

更偏技术的业务指标

库存满足率

用户搜索后有可售库存的占比。低于90%说明供应链覆盖不足或缓存过期。

查询响应时间

机票/酒店搜索结果返回时间。超过3秒用户流失率显著上升。

出票成功率

支付成功到最终出票的比例。涉及供应商稳定性、订单状态同步、异常处理。

客单价 (AOV)

平均每单金额。受推荐策略、打包能力、交叉销售影响。

30日留存率

注册/首单用户30天后仍活跃的比例。反映产品价值和推荐质量。

NPS

净推荐值。客服响应速度、退改签体验、行程变更处理能力都会影响。

09出行行业的6种商业模式

同一个出行产品，可以有完全不同的变现方式。理解商业模式，有助于你理解系统为什么要这样设计。

佣金模式

按交易金额抽取一定比例。OTA、B2B批发商的主要模式。
技术重点：订单追踪、佣金计算、对账结算

差价模式

低价买断库存，高价卖出。包房、机票切位常用。
技术重点：库存锁定、价格弹性分析、风险控制

SaaS订阅

企业客户按月/年付费使用差旅管理系统。
技术重点：多租户架构、权限管理、企业级SLA

API调用费

按API调用次数收费，或按成功交易量抽成。
技术重点：计费系统、限流、SLA监控

广告模式

向供应商出售搜索结果的优先展示位。
技术重点：竞价排序、广告相关性、点击率优化

会员增值

付费会员享受专属价格、退改签特权。
技术重点：权益系统、价格歧视、会员生命周期管理

10案例：一次完整出行的商业流转

假设用户小明通过龙虾出行APP预订了一次「北京→上海」的商务出行。让我们看看钱和数据是怎么流动的。

1
搜索与推荐

小明输入「周五下午北京飞上海」。系统调用GDS/NDC/酒店API查询库存，AI推荐引擎根据偏好排序。
商业价值：推荐质量决定转化率。

2
选择产品与下单

小明选择机票+酒店打包方案。系统校验库存、计算价格（含佣金、服务费、税费）。
商业价值：打包提升客单价33%。

3
支付与风控

小明支付¥2,800。风控系统实时判断交易风险，支付网关处理资金。
商业价值：拒付率每降0.1%，省数十万。

4
出票与确认

系统向航司下单出票，向酒店发送预订请求，拿到PNR和确认号，同步给小明。
商业价值：出票成功率必须接近99%。

5
履约与服务

出行当天，系统推送登机提醒、酒店入住码。如遇航班延误，AI客服主动介入改签。
商业价值：服务质量决定NPS和复购。

6
结算与分账

平台扣除佣金后，向航司结算机票款、向酒店结算房费、向接送车供应商结算车费。
商业价值：分账准确直接影响供应商关系和财务合规。

12技术团队的行动清单

课程结束，以下是每位技术同学可以立刻践行的3件事。

1. 读一遍产品PRD

不要只看技术实现部分，重点关注「商业价值」和「成功指标」章节。问问自己：这个功能解决谁的什么问题？如何衡量成功？

13Q&A 与讨论

有任何问题，现在就是提问的最佳时机。以下是几个常见的「好问题」供参考。

❓ 为什么机票价格每秒都在变？

因为航司的收益管理系统根据需求预测、竞品价格、剩余座位实时调整。缓存策略需要平衡「新鲜度」和「查询成本」。

❓ 为什么酒店要签「价格一致性协议」？

酒店希望避免不同渠道价格差异过大导致用户信任崩塌。技术上需要监控各渠道价格，防止违规低价。

❓ 打车业务的「动态加价」是怎么算的？

基于供需比、预计等待时间、取消率等因子，通过算法模型实时计算。目标是平衡供给和需求，而不是单纯涨价。

❓ 作为后端工程师，我怎么理解「转化率」？

转化率 = 完成目标行为的用户数 ÷ 总用户数。你负责的接口性能、稳定性、错误处理都会直接或间接影响它。
